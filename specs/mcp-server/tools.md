# MCP Server: Tools Specification

## Naming Convention

All tools follow: `tasks_{intent}` — intent-based, not data-surface-based.

Examples: `tasks_capture`, `tasks_review`, `tasks_modify`

---

## Common Parameters

Every tool accepts these parameters:

| Field | Type | Default | Description |
|---|---|---|---|
| `user_id` | string | `"default"` | User identifier for data scoping. Agent passes from conversation context. No auth validation. |

---

## Session Initialization

Every request (except `initialize`) requires a valid session. The MCP Streamable
HTTP handshake is:

1. Client sends `initialize` → server responds with `Mcp-Session-Id` header
2. Client sends `notifications/initialized` → server acknowledges (202)
3. All subsequent calls include `Mcp-Session-Id: <session>` header

Session IDs are opaque strings generated by the server.

## Annotations Template

Every tool includes explicit annotations using `ToolAnnotations` dataclass:

```python
from mcp.types import ToolAnnotations

annotations=ToolAnnotations(
    readOnlyHint=False,      # True if tool does not mutate state
    destructiveHint=False,   # True if tool can delete/destroy
    idempotentHint=False,    # True if repeated calls have no additional effect
    openWorldHint=False,     # True if tool interacts with external entities
)
```

---

## Response Format

Every data-returning tool accepts an optional `response_format` parameter:

| Value | Behavior |
|---|---|
| `"json"` (default) | Structured JSON, all fields and metadata included |
| `"markdown"` | Human-readable formatted text, concise, timestamps readable |

---

## Pagination Pattern (for list/search tools)

Input:
| Field | Type | Default | Description |
|---|---|---|---|
| `limit` | int | 20 | Results per page (1-100) |
| `offset` | int | 0 | Number of results to skip |

Output:
| Field | Type | Description |
|---|---|---|
| `total` | int | Total matching results |
| `count` | int | Results in this page |
| `offset` | int | Current offset |
| `has_more` | bool | More results available |
| `next_offset` | int or null | Offset for next page |
| `items` | list | Page results |

---

## Error Handling Contract

Every tool returns errors in a uniform format:

```
Error: <what went wrong>. <suggested next step>.
```

Examples:
- `Error: Item not found. Check the item_id and try again.`
- `Error: Rate limit exceeded. Wait before making more requests.`
- `Error: Request timed out. Please retry.`

Errors are returned as text content in the result (with `isError: false` at the protocol level — the error message is embedded in the text, not as a protocol error).

---

## Character Limit

Tools that return lists or search results enforce a 25K character limit on response size.

If exceeded:
- Results are truncated
- `truncated: true` is set
- `truncation_hint: "Response truncated. Use limit/offset or add filters to narrow results."` is appended

---

## Time Format

- **Server stores and returns all timestamps in UTC** (ISO 8601).
- Every tool that accepts or returns time data supports an optional `timezone` parameter (IANA timezone, e.g. `"America/New_York"`).
- When `timezone` is provided, Markdown responses render timestamps in that timezone.
- JSON responses always return UTC — the agent is responsible for converting to the user's timezone for display.
- **Agent contract**: The agent must convert the user's natural language time to UTC before calling the server, and convert UTC back to the user's timezone when presenting results.

---

## Idempotency

Mutating tools accept an optional `idempotency_key` field (UUID string).

If provided and the server has already processed a request with the same key within the TTL window, the original response is returned instead of re-executing.

---

## Tool Inventory

All tools operate on a unified "work item" model with a `type` field (task, appointment, reminder). There is no separate CRUD per domain — the agent chooses the right intent, and the server handles schema internally.

| Tool | Intent | ReadOnly | Destructive | Idempotent | OpenWorld | Description |
|---|---|---|---|---|---|---|
| `tasks_capture` | "Remind me to call mom tomorrow at 5" | No | No | Yes | No | Create any work item with optional reminder, due date, priority, type, all in one call |
| `tasks_review` | "What's on my plate today?" / "What's overdue?" | **Yes** | No | Yes | No | Query work items — list by status, date range, search text, overdue, today's agenda. Supports pagination. |
| `tasks_modify` | "Move the dentist to Wednesday and rename it" | No | No | Yes | No | Update any field of any item — rename, reschedule, reprioritize, change type |
| `tasks_resolve` | "I finished the report" / "Skip the gym" | No | No | Yes | No | Mark item complete, cancel, skip — terminal status changes |
| `tasks_remove` | "Delete that grocery task — I don't need it" | No | **Yes** | Yes | No | Permanently delete a work item. `destructiveHint` annotation signals need for user confirmation at agent level. |
